# Headroom Upgrade and Output Shaping Design

## Goal

Upgrade the persistent local Headroom deployment from 0.30.0 to 0.31.0 and enable measurable output-token shaping without disrupting its existing Claude, Codex, memory, and tool-result behavior.

## Current State

- `headroom-ai` 0.30.0 is installed as a global `uv` tool.
- The `default` launchd deployment runs on `127.0.0.1:8787`.
- Claude and Codex route through the persistent proxy.
- Persistent memory and Read maturation are enabled.
- Tool results are protected and CCR marker/tool injection is disabled.
- Output shaping and its measurement holdout are disabled.

## Selected Approach

Use Headroom's learned verbosity policy with a 10% unshaped control group. Learning across all available Claude and Codex projects supplies a representative baseline. The control group lets `headroom output-savings` report measured savings instead of relying only on a counterfactual estimate.

Fixed verbosity is rejected because it ignores established usage behavior. Continuous verbosity autotuning is rejected because it changes the policy during the measurement window and makes the first comparison harder to interpret.

## Configuration

The persistent `default` deployment will add these runtime values:

```text
HEADROOM_OUTPUT_SHAPER=1
HEADROOM_OUTPUT_HOLDOUT=0.1
```

The learned verbosity level will be generated by:

```sh
headroom learn --verbosity --apply --all
```

Existing routing, memory, Read maturation, protected tool-result, HTTP, retry, and telemetry values must remain unchanged unless Headroom 0.31.0 performs a documented schema migration.

## Upgrade Flow

1. Capture the current package version, `headroom perf` baseline, deployment health, manifest, and generated launch script.
2. Back up the manifest and generated launch script outside the repository.
3. Upgrade only `headroom-ai` through `uv`; do not upgrade unrelated tools.
4. Confirm the new CLI reports version 0.31.0 before changing the service.
5. Learn and persist the preferred verbosity across all discovered projects.
6. Add the shaper and holdout values to the durable deployment configuration and generated service environment.
7. Restart `com.headroom.default` once through Headroom's managed restart command.

## Failure Handling and Rollback

If the package upgrade fails, leave the running 0.30.0 service unchanged. If learning fails, do not enable shaping. If the restarted 0.31.0 deployment fails health or routing checks, restore the backed-up deployment files, reinstall `headroom-ai==0.30.0`, and restart the managed deployment.

No manual process kill is part of the normal flow. This limits interruption to the single managed restart and avoids terminating the proxy while it is serving the active agent session.

## Verification

Completion requires fresh evidence for all of the following:

- `headroom --version` reports 0.31.0.
- `headroom doctor --json` reports a healthy proxy, deployment, Claude route, and Codex route.
- launchd reports `com.headroom.default` running.
- The live service environment contains `HEADROOM_OUTPUT_SHAPER=1` and `HEADROOM_OUTPUT_HOLDOUT=0.1`.
- Existing memory, Read maturation, protected tool-result, routing, and telemetry configuration remains present.
- `headroom output-savings` recognizes the learned baseline; measured savings may require subsequent shaped and holdout traffic before a confidence interval is available.
- `headroom perf --hours 72` remains available as the pre-change comparison window.

## Scope

This change does not alter Headroom source code, enable verbosity autotuning, change the 10% holdout, prune memory, or tune compression ratios. Those changes require separate evidence after output-shaping data has accumulated.
